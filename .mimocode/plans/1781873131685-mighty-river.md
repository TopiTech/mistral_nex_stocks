# Mistral NeX Stocks - 包括的コードレビュー

## プロジェクト概要
Flask + Chrome Extension + Native Host で構成されたAI搭載株式投資ダッシュボード。
Mistral AI / DuckDuckGo / LangSearch を活用した市場分析、リアルタイムSSE配信、ポートフォリオ管理機能を持つ。

---

## 1. アーキテクチャ評価

### 良好な点
- **モジュール分割が優秀**: `app_state.py`, `app_helpers.py`, `app_bg.py`, `route_helpers.py` に適切に分離。`routes/`, `services/`, `utils/` のパッケージ構造も清潔
- **Circuit Breaker Pattern**: yfinance, Mistral, LangSearch にサーキットブレーカーを実装。外部API障害時のフォールバックが適切
- **Structued Outputs (Pydantic)**: Mistral APIのStructured Outputを`StockAnalysis`, `NewsSummaryModel`で活用。型安全なJSON生成
- **SSE + ポーリング二重構成**: ストリーミング切断時の60秒フォールバックが安定性に貢献
- **セキュリティ層が厚い**: CSP (Talisman), CSRF, Rate Limiting, Sec-Fetch-Site チェック, shutdown token (single-use), レスポンスヘッダのサニタイズ

### 改善可能な点

#### 1.1 app.py が依然として989行と巨大
**重要度: 中**
`app.py` にはまだ `_get_or_create_shutdown_token`, `_consume_shutdown_token`, `_rotate_shutdown_token`, `schedule_news_warmup`, `_filter_recent_market_news_items` 等の関数が残存。Blueprint分割済みルートに移行済みだが、these utility functions could be extracted to a dedicated module (e.g. `shutdown_utils.py`, `news_helpers.py`).

#### 1.2 ルートファイル（api_analysis.py, api_stocks.py, api_system.py, pages.py）のインポート重複
**重要度: 低-中**
4つのルートBlueprintでほぼ同一の60行以上に及ぶimportブロックが繰り返されている。共通importを `routes/__init__.py` に集約すべき。

#### 1.3 AppState の動的ディスパッチ（__getattr__）
**重要度: 中**
`AppState.__getattr__` で属性を動的にグループに委譲する設計は柔軟だが、IDEの補完が効かず、デバッグ時に属性の所在が追いにくい。明示的なプロキシメソッドの追加を推奨。

## 2. UI/UX 評価

### 良好な点
- **ダークテーマのグラスモーフィズム**: `backdrop-filter: blur()` を活用したモダンなUI
- **スケルトンUI**: 初回ロード時のローディング状態が視覚的に良好
- **スパークライン**: カード内にミニチャートを表示し、一目でトレンドが把握可能
- **色覚多様性配慮**: `▲`/`▼` 記号で色だけでなく記号でも増減を伝える（`ui.js:81-84`）
- **SSE接続状態表示**: Live Streaming/Reconnecting/Paused の状態がヘッダーに表示
- **レスポンシブデザイン**: `@media` で759px, 760px, 1180px, 2100px の4段階ブレークポイント

### 改善可能な点

#### 2.1 モバイル対応の不備
**重要度: 中**
- `stocks-list` の `minmax(360px, 1fr)` は360px未満の端末（iPhone SE等）で横スクロール発生
- ヘッダーの `flex-wrap: wrap` は良いが、タブバーが横スクロールになるためスクロール操作が直感的でない
- ヒートマップページ（heatmap.html）のレスポンシブ対応を確認推奨

#### 2.2 ニュースセクションの初期状態
**重要度: 低**
初期表示が「下部の「リアルタイム更新」で最新 of AI要約を取得します」と固定テキスト。自動取得を検討すべきか、少なくとも「取得するにはボタンをクリック」と明示すべき。

#### 2.3 アラート機能の UX
**重要度: 低-中**
- アラート設定は `localStorage` のみで保持。ブラウザをクリアすると消失
- ブラウザ通知の `Notification.permission` は初期訪問時のみリクエスト。設定ページからの再許可方法がない
- MA5クロス通知の精度が低い（最新の1点のみで判定）

#### 2.4 チャットUI
**重要度: 低**
- チャット履歴がページリロードで消える（バックエンド側の `chat_history` はLRUで50エントリ制限）
- チャットログの高さが固定150px（`chat-log { height: 150px }`）。レスポンスが長い場合に見づらい

## 3. バックエンド評価

### 良好な点
- **エラーハンドリング**: グローバルエラーハンドラ（400, 403, 404, 405, 413, 429, 500）が完備
- **レート制限**: `@rate_limit` デコレータでエンドポイントごとに制限
- **キャッシュ**: `TTLCache` + negative cache でLLM呼び出しを最適化
- **yfinance対策**: UAローテーション, curl_cffi によるTLS偽装, サーキットブレーカー, 指数バックオフ
- **構造化ログ**: JSONロガー + テキストフォーマッターの二重出力
- **グレースフルシャットダウン**: ワンタイムshutdown token + プロセス終了処理

### 改善可能な点

#### 3.1 同期的 `time.sleep()` の使用
**重要度: 中**
`ai_service.py:382` で `time.sleep(wait_before)` がリクエストスレッド内で実行される。Flask の threaded=True ではリクエストスレッドを消費するため、高負荷時にスレッド枯渇リスクがある。`asyncio` またはバックグラウンドキューへの移行を検討。

#### 3.2 Mistralクライアントのスレッドキー
**重要度: 低**
`app_state.py:717` で `(api_key, thread_id)` をキーにクライアントをキャッシュ。`threading.get_ident()` はスレッドプールでは再利用されるため、意図通りに動作するが、 ThreadPoolExecutor のスレッド再利用時に古いクライアントが残る可能性。LRU缓存は maxsize=128 で制限済み。

#### 3.3 チャット履歴のLRU管理
**重要度: 低**
`api_analysis.py:264-283` で `OrderedDict` を使った手動LRU。`functools.lru_cache` や `cachetools.LRUCache` で簡略化可能。また、system prompt のみを保持する設計だが、10件に制限後最初のメッセージ（system prompt）を復元するロジックが冗長。

#### 3.4 ニュースAPIのタイムアウト処理
**重要度: 中**
`api_analysis.py:425-449` で `concurrent.futures.wait(timeout=NEWS_CONTEXT_WAIT_TIMEOUT)` 使用。timeout 後に `fut.cancel()` するが、実行中のスレッドプールタスクはキャンセルされない（PythonのThreadPoolExecutorの制約）。実質的にタスクは完了するまで残る。

#### 3.5 セキュリティ: `_is_local_request` の信頼性
**重要度: 低-中**
ローカルホストからのリクエストのみ許可する設計だが、`_is_local_request` が `request.remote_addr` を使用。プロキシ経由の場合は信頼性が低下。フロントエンドからの呼び出しのみで使用されるため実用上問題ないが、CSPやCORSとの二重防御は適切。

## 4. セキュリティ評価

### 良好な点
- **CSP (Content Security Policy)**: Talismanによる厳格なポリシー。`'unsafe-inline'` は排除し nonce を使用
- **CSRF保護**: Flask-WTF のCSRFProtect + POST リクエストでの Sec-Fetch-Site チェック
- **shutdown token**: ワンタイムトークン + トークンローテーション
- **APIキー保護**: keyring/DPAPI による安全な保存。`MNS_ALLOW_PLAINTEXT_SECRETS` でのフォールバック時に重大警告
- **入力検証**: フロントエンド/バックエンド両方でシンボル、数値、テキストのサニタイズ
- **XSS対策**: DOM API (textContent) の使用推奨。`innerHTML` は使わない（`chart.js:247` の `createEl` ヘルパー）

### 改善可能な点

#### 4.1 `config.json` へのAPIキー保存時のファイルパーミッション
**重要度: 中**
`config_utils.py` で `enforce_secure_permissions` を使用しているが、Windows での ACL 設定が限定的。`chmod 0o600` は Unix のみ有効。Windows では `SetNamedSecurityInfo` API による ACL 設定を確認推奨。

#### 4.2 フロントエンドでのAPIキー送信
**重要度: 低**
APIキーを `Authorization: Bearer` ヘッダーで送信。CSP の `connect-src` で `https://api.mistral.ai` を許可済み。ローカルホストでのみ使用されるためリスクは低いが、localStorage からの移行が完了したことを確認推奨。

#### 4.3 CSP Report-Only モード
**重要度: 低**
`CSP_ENFORCE=false` で Report-Only モード。開発環境では合理的だが、本番環境での強制化が必須。

## 5. パフォーマンス評価

### 良好な点
- **フロントエンドのDOM操作最適化**: `documentFragment` の使用, `textContent` での更新, `WeakMap` によるチャートインスタンス管理
- **変更検出**: `computeStockHash` でSSE毎の不要なDOM更新をスキップ
- **スパークラインのビューポート判定**: `isElementInViewport` でスクロール外の描画をスキップ
- **プリフェッチ**: チャート展開前のヒストリカルデータプリフェッチ
- **デバウンス**: ポートフォリオ描画のデバウンス（500ms）

### 改善可能な点

#### 5.1 SSE のハートビート間隔
**重要度: 低**
サーバー側: 15秒、クライアント側: 60秒タイムアウト。この設定は適切。しかし、タブが非表示の場合は30秒後に切断し、復帰時に再接続。この設計は良好。

#### 5.2 チャート再描画の頻度
**重要度: 低**
`updateStockUI` でSSEメッセージごとにスパークラインとメインチャートの更新判定を行う。`shouldUpdateSparkline` でシグネチャベースの重複防止があるが、メインチャートの更新判定は `period === "3mo" || period === "1d"` のみ。他の期間の場合は最終ポイントのみ更新。

#### 5.3 ヒストリカルデータの取得
**重要度: 低-中**
1日の場合のみ5分足で取得し、失敗時に日足にフォールバック。MA25計算のために拡張期間で取得後トリミング。この設計は合理的だが、`extended_period_map` のマッピングが密結合。

## 7. コード品質・保守性

### 良好な点
- **pylint/ruff の使用**: `pyproject.toml` でリンチルール設定
- **型ヒント**: 主要なパラメータに型ヒント付き（`typing` モジュール使用）
- **リージョナルコメント**: `# #region` / `# #endregion` でコードブロックを整理
- **定数の一元管理**: `constants.py` で環境変数経由のチューニングパラメータを一元管理

### 改善可能な点

#### 6.1 JavaScript のモジュール化
**重要度: 中**
11個のJSファイルが `<script>` タグで直列読み込み。ES Modules への移行を検討すべきか、少なくとも読み込み順序の依存関係を明示すべき。

#### 6.2 CSS の重複
**重要度: 低**
`index.css` (2882行) が非常に巨大。コンポーネントごとのCSS分割を検討。

## 8. 推奨改善アクション（優先度順）

1. **ルートファイルの共通import集約** - 各Blueprintの冒頭60行の重複importを `routes/__init__.py` に集約
2. **app.py の更なる分割** - shutdown関数を `shutdown_utils.py` に、ニュース関数を `news_helpers.py` に分離
3. **モバイルCSS修正** - `minmax(360px, 1fr)` を `minmax(min(360px, 100%), 1fr)` に変更
4. **JavaScript の設定ページ分離** - `settings.js` で `DEFAULT_SYMBOLS` と `APP_CONFIG` が `state.js` と重複定義されているのを解消
5. **テストカバレッジ向上** - ヒートマップ、設定ページ、チャット機能のテスト追加
